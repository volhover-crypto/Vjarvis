# Code Review and Quality (Python/Bash Edition)

## Overview

Многоосевой код-ревью с quality gates. Каждое изменение проверяется перед мержем — без исключений. Пять осей: корректность, читаемость, архитектура, безопасность, производительность.

**Стандарт одобрения:** Одобряй изменение, если оно определённо улучшает здоровье кода, даже если не идеально. Идеального кода не существует — цель непрерывное улучшение.

## Пять осей ревью

### 1. Корректность

- Соответствует ли код спецификации/задаче?
- Обработаны ли edge cases (None, пустые значения, граничные значения)?
- Обработаны ли ошибки (не только happy path)?
- Проходят ли тесты? Тесты проверяют правильные вещи?
- Нет ли off-by-one, race conditions, несогласованности состояния?

**Python-специфичные проверки:**
- Корректная работа с None (не путать с 0/пустой строкой)
- Правильное использование менеджеров контекста (`with`)
- Корректная обработка исключений (не голые `except:`)
- Правильная работа с изменяемыми аргументами по умолчанию

**Bash-специфичные проверки:**
- Все переменные в кавычках (`"$var"`)
- Проверка кодов возврата (`set -e` или `|| exit`)
- Нет незакавыченных путей с пробелами

### 2. Читаемость и простота

- Имена описательные и соответствуют конвенциям? (нет `tmp`, `data`, `result` без контекста)
- Поток управления прямой (избегать вложенных тернарных операторов)
- Код организован логически (связанный код сгруппирован)
- Можно ли сделать короче? (1000 строк вместо 100 — провал)
- Абстракции оправдывают свою сложность? (не обобщать до третьего случая)

**Python-специфичные:**
- PEP 8 соблюдён
- Type hints где уместно
- Docstrings для публичных функций
- Нет мёртвого кода: неиспользуемые импорты, переменные `_unused`

### 3. Архитектура

- Следует ли изменение существующим паттернам?
- Чистые границы модулей?
- Нет ли дублирования кода?
- Зависимости текут в правильном направлении (нет циклических)?
- Уровень абстракции подходящий?

**Для наших проектов:**
- IoT: чёткое разделение слоёв (сбор данных → обработка → хранение → представление)
- Агрономия: разделение доменной логики и инфраструктуры
- OpenClaw-навыки: SKILL.md + scripts/ + references/

### 4. Безопасность

- Пользовательский ввод валидирован и санитизирован?
- Секреты не в коде, логах, git?
- SQL-запросы параметризованы (нет конкатенации строк)?
- Зависимости из доверенных источников, без известных уязвимостей?
- Внешние данные считаются недоверенными?

**Python-специфичные:**
- Нет `eval()` / `exec()` с пользовательским вводом
- Нет `pickle.loads()` с недоверенными данными
- `subprocess` с `shell=True` — только если необходимо и с экранированием

**Bash-специфичные:**
- Нет `eval` с пользовательским вводом
- Все внешние данные экранированы
- Файлы создаются с безопасными правами

### 5. Производительность

- Нет ли N+1 query patterns?
- Нет ли неограниченных циклов?
- Нет ли больших объектов в hot paths?
- Пагинация на list endpoints?

**Python-специфичные:**
- Генераторы вместо списков для больших данных
- `lru_cache` для повторяющихся вычислений
- Профилирование перед оптимизацией (`cProfile`, `line_profiler`)

## Размер изменений

```
~100 строк → Хорошо. Обозримо за один присест.
~300 строк → Допустимо, если одно логическое изменение.
~1000 строк → Слишком большой. Раздели.
```

**Стратегии разделения:**
- **Stack** — маленькое изменение, следующее на его основе
- **По группам файлов** — отдельные изменения для разных ревьюеров
- **Горизонтально** — общий код/заглушки сначала, потом потребители
- **Вертикально** — маленькие full-stack срезы фичи

**Отделяй рефакторинг от новой функциональности.**

## Процесс ревью

### Шаг 1: Пойми контекст
```
- Что это изменение пытается сделать?
- Какую спецификацию/задачу реализует?
- Какое ожидаемое изменение поведения?
```

### Шаг 2: Сначала тесты
```
- Тесты существуют?
- Тестируют поведение (не детали реализации)?
- Edge cases покрыты?
- Имена тестов описательные?
- Тесты поймают регрессию?
```

### Шаг 3: Ревью реализации
Для каждого изменённого файла:
```
1. Корректность: код делает то, что тесты говорят?
2. Читаемость: понимаю ли я без помощи?
3. Архитектура: вписывается ли в систему?
4. Безопасность: уязвимости?
5. Производительность: бутылочные горлышки?
```

### Шаг 4: Категоризируй находки

| Префикс | Значение | Действие автора |
|---------|----------|-----------------|
| *(нет)* | Обязательное изменение | Должно быть исправлено |
| **Critical:** | Блокирует мерж | Уязвимость, потеря данных, сломанная функциональность |
| **Nit:** | Мелочь, опционально | Автор может проигнорировать |
| **Optional:** / **Consider:** | Предложение | Стоит рассмотреть |
| **FYI** | Информация | Действий не требуется |

### Шаг 5: Верификация
```
- Какие тесты запущены?
- Билд прошёл?
- Ручное тестирование сделано?
- Есть before/after сравнение?
```

## Multi-Model Review Pattern

```
Модель A пишет код
    │
    ▼
Модель B ревьюит корректность и архитектуру
    │
    ▼
Модель A исправляет
    │
    ▼
Человек принимает финальное решение
```

## Dead Code Hygiene

После рефакторинга проверяй:
1. Недостижимый или неиспользуемый код
2. Перечисли явно
3. **Спроси перед удалением:** "Удалить эти неиспользуемые элементы: [список]?"

## Честность в ревью

- **Не рубберстик.** "LGTM" без доказательств ревью никому не помогает.
- **Не смягчай реальные проблемы.** Баг, который попадёт в production — это не "минорная проблема".
- **Квантифицируй.** "Этот N+1 запрос добавит ~50ms на элемент" лучше, чем "это может быть медленно".
- **Не будь сикофантом.** Если реализация имеет проблемы, скажи прямо и предложи альтернативы.

## Дисциплина зависимостей

**Перед добавлением зависимости:**
1. Решает ли существующий стек эту задачу?
2. Насколько большая зависимость?
3. Активно ли поддерживается?
4. Есть ли известные уязвимости?
5. Совместима ли лицензия?

**Правило:** Предпочитай стандартную библиотеку и существующие утилиты новым зависимостям. Каждая зависимость — обязательство.

## Checklist

```markdown
## Review: [Название изменения]

### Контекст
- [ ] Понимаю, что делает изменение и почему

### Корректность
- [ ] Соответствует спецификации/задаче
- [ ] Edge cases обработаны
- [ ] Ошибки обработаны
- [ ] Тесты покрывают изменение

### Читаемость
- [ ] Имена ясные и консистентные
- [ ] Логика прямая
- [ ] Нет излишней сложности

### Архитектура
- [ ] Следует существующим паттернам
- [ ] Нет лишней связанности
- [ ] Подходящий уровень абстракции

### Безопасность
- [ ] Нет секретов в коде
- [ ] Входные данные валидированы
- [ ] Нет injection уязвимостей
- [ ] Auth checks на месте

### Производительность
- [ ] Нет N+1 паттернов
- [ ] Нет неограниченных операций

### Верификация
- [ ] Тесты проходят
- [ ] Билд успешен

### Вердикт
- [ ] **Approve** — Готово к мержу
- [ ] **Request changes** — Нужны исправления
```

## Red Flags

- Код мержится без ревью
- Ревью проверяет только тесты (игнорируя другие оси)
- "LGTM" без доказательств реального ревью
- Большие PR, которые "слишком велики для ревью" (раздели)
- Нет регрессионных тестов с фиксами багов
- Комментарии без severity labels

# Harness Engineering (OpenClaw Edition)

## Overview
Harness engineering — это проектирование «обвязки» вокруг агента: тесты, линтеры, архитектурные ограничения, которые дают модели чёткую, невероятностную обратную связь. Это снижает галлюцинации, усиливает правильные паттерны и делает поведение агента более предсказуемым.

## Когда использовать
- При начале работы над новым модулем/навыком.
- Когда замечаешь, что агент начинает «галлюцинировать» или отклоняться от установленных конвенций.
- Перед запуском сложного multi‑step workflow (например, code review, анализ логов).
- При обновлении контекста (добавлении новых навыков/файлов) – проверять, не ломает ли это существующие ограничения.

## Интеграция с OpenClaw
1. **Структурные тесты** – разместить в `tests/` проекта и добавить скрипт `run_structural_tests.sh`, который выводит короткое саммари (пройдено/провалено, ключевые нарушения).  
2. **Кастомные линтеры** – можно использовать готовые (shellcheck, pylint, flake8) либо написать свои правила (например, проверка, что все новые навыки имеют `SKILL.md` и `scripts/`).  
3. **Архитектурные ограничения** – фиксировать в файле `ARCHITECTURE.md` (или в `vault/WIKI.md`) правила вроде:  
   - «Все новые навыки должны lazy‑load тяжелые ресурсы».  
   - «Subagent‑сессии не должны возвращать более 2000 токенов в основной контекст».  
   Скрипт проверки будет читать эти правила и сообщать о нарушениях.  
4. **Обратная связь в контекст** – вывод линтеров и тестов добавлять в `vault/learning-log.md` с тегом `[harness]` и, при необходимости, в `MEMORY.md` как краткое предупреждение.

## Чек‑лист перед запуском задачи
- [ ] Есть ли актуальный `ARCHITECTURE.md`/`WIKI.md` с ограничениями?  
- [ ] Запущены ли структурные тесты (скрипт возвращает 0)?  
- [ ] Запущены ли кастомные линтеры (уровень предупреждений ≤ порог)?  
- [ ] Вывод линтеров/тестов записан в журнал обучения?  
- [ ] При нарушении – агент получает понятное сообщение, а не просто «ошибка».

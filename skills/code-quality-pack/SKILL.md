# Code Quality Pack

Комплексный набор из 4 проверок качества кода. Применяется перед коммитом, при ревью и по запросу.

## Когда использовать

- Перед коммитом любого нетривиального изменения
- При ревью кода (свой или чужой)
- По запросу: «проверь качество», «почисти код», «верь что это работает»
- После генерации кода AI — обязательно

---

## 1. Deslop — Удаление AI-слабины

**Цель:** убрать артефакты, которые AI-модели добавляют по привычке.

### Что искать и удалять

| Паттерн | Пример | Что делать |
|---------|--------|------------|
| Лишние комментарии | `# increment counter` рядом с `i += 1` | Удалить |
| Избыточные try/catch | `try: x = int(y) except: x = 0` в trusted path | Упростить |
| Приведения к `any`/`object` | `cast(value, any)` чтобы обойти типы | Исправить типы |
| Глубокая вложенность | 4+ уровня if/for | Вынести в функции, early return |
| Защитные проверки везде | `if x is not None` перед каждым обращением | Оставить только на границах |
| Мёртвый код | Неиспользуемые импорты, переменные `_unused` | Удалить |
| Шаблонные ответы | `"""This function does X."""` без полезной информации | Удалить или дополнить |

### Guardrails

- Не менять поведение — только чистить стиль
- Минимальные правки, не переписывать всё
- Если не уверен — спросить, не удалять

## 2. Compiler Check — Проверка компиляции и типов

**Цель:** убедиться, что код компилируется и проходит проверку типов.

### Workflow

1. Определить тип проекта и команды проверки:

| Тип проекта | Команды |
|-------------|---------|
| Python | `python -m py_compile <file>` или `mypy <file>` или `ruff check <file>` |
| TypeScript | `npx tsc --noEmit` |
| Bash | `bash -n <file>` (syntax check) |
| Rust | `cargo check` |
| Go | `go vet ./...` |
| C/C++ | `gcc -fsyntax-only <file>` |

2. Запустить проверку, собрать ошибки
3. Группировать по файлу и типу ошибки
4. Исправить самые критичные сначала
5. Перезапустить проверку до чистого результата

### Формат вывода

```
Compile Check: [PASS / FAIL]

Errors (N):
- <file>:<line> — <error type>: <message>

Fixes applied:
- <file> — <what was fixed>

Remaining blockers:
- <file> — <unresolved issue>
```

## 3. Thermo Review — Строгий аудит поддерживаемости

**Цель:** глубокая проверка архитектурного какчества. Применять для изменений >100 строк или при добавлении новых модулей.

### Принципы

**0. Амбициозность упрощения.** Не «можно чуть чище» — ищи code judo: реструктуризацию, которая удаляет целые категории сложности.

**1. Правило 1000 строк.** Не допускай перехода файла за 1000 строк без веской причины. Выделяй хелперы, подмодули, абстракции.

**2. Анти-спагетти.** Подозрительны: ad-hoc условия, разбросанные special cases, one-off ветки в чужих потоках. Выноси логику в выделенные абстракции.

**3. Дизайн > «работает».** Если поведение можно сохранить, а структуру сделать чинее — делай чище. Не резинстуй «оно работает».

**4. Прямой код > магия.** Подозрительны: brittle ad-hoc механизмы, generic-обработка, скрывающая простую структуру. Тонкие обёртки без ясности.

**5. Чистые границы типов.** Подозрительны: `any`, `unknown`, избыточная optionality, cast-heavy код. Предпочитай явные типы и контракты.

**6. Канонический слой.** Логика должна жить в правильном модуле. Не допускай протечки feature-логики в shared paths.

**7. Оркестрация.** Независимая работа не должна быть сериализована без причины. Связанные обновления должны быть атомарными.

### Вопросы для каждого изменения

- Есть ли code judo, которое сделает это драматически проще?
- Улучшает ли это локальную архитектуру или ухудшает?
- Не протекла ли логика через границу модуля?
- Не вырос ли файл за здоровый предел?
- Нет ли повторяющихся условий, сигнализирующих о пропущенной абстракции?
- Эта абстракция оправдывает себя или просто обёртка?

### Формат вывода

```
Thermo Review: [PASS / WARN / FAIL]

Structural findings:
- [CRITICAL] <file> — <issue>
- [WARN] <file> — <suggestion>

Size analysis:
- <file>: <N> lines (<status>)

Verdict: <краткое резюме 1-2 предложения>
```

## 4. Verify This — Верификация claims

**Цель:** доказать или опровергнуть конкретное claim через воспроизводимые артефакты.

### Когда использовать

- «Проверь что это работает»
- «Докажи что фикс помог»
- «Покажи что поведение изменилось»
- Любое утверждение, которое можно измерить

### Workflow

1. **Переформулировать claim** в фальсифицируемую форму: условие + метрика + порог
2. **Выбрать минимальную поверхность** для проверки (тест, скрипт, HTTP-запрос)
3. **Capture baseline** — состояние ДО (старый коммит, предыдущая версия)
4. **Capture treatment** — состояние ПОСЛЕ (текущий код)
5. **Сравнить артефакты** — числа, скриншоты, вывод терминала, HTTP-ответы
6. **Вердикт:** `VERIFIED` | `NOT VERIFIED` | `INCONCLUSIVE`

### Формат вывода

```
VERIFIED | NOT VERIFIED | INCONCLUSIVE
Claim: <фальсифицируемый claim>

Evidence:
<metric>: baseline=<...>, treatment=<...>, delta=<...>, threshold=<...>

Reasoning:
<одно предложение: что показывают данные и есть ли конфаундеры>
```

### Guardrails

- Не смягчай негативный результат. Чёткий `NOT VERIFIED` полезен.
- Не используй для расплывчатых claims вроде «код чище» — попроси измеримый критерий
- Минимальные артефакты — только то, что нужно для вердикта

## Рекомендуемый порядок применения

```
1. Deslop      → быстро, перед каждым коммитом
2. Compiler    → быстро, перед каждым коммитом
3. Thermo      → для изменений >100 строк или новых модулей
4. Verify      → по запросу или при фиксации багов
```
